## 背景

Lumina Note 当前是一个 local-first 的桌面知识工具。现有的移动端方向不应继续被理解为“必须做完整原生客户端”。

在记录基准提交 `21ba54052b698494538f74fa1bcd3660a8ecec45` 下，仓库里的 `mobile/` 目录包含早期的 iOS SwiftUI 和 Android Kotlin/Jetpack Compose companion app 尝试。这个方向低估了开发和维护成本：继续维护两套原生客户端，会把产品拖向“移动版完整 Lumina Note”，而这不是当前最低成本、最高价值的路径。

因此，基于该提交状态下的 `mobile/` 代码，可以视为可废弃的历史实现。它不再作为后续移动端主线，也不应作为新移动产品形态的架构基础。

当前真正需要回答的问题不是：

> 怎么做一个手机版 Lumina Note？

而是：

> 用最低成本，做出一个在桌面之外也能发挥 Lumina AI Agent 价值的移动与云端形态。

## 核心判断

移动端不应该是强编辑器。

Lumina 的主要交互是 AI，而且这个 AI 不是普通聊天框，而是能围绕用户知识库工作的 Agent。手机端最有价值的场景不是长文写作、复杂 Markdown 编辑、block 操作、知识图谱或 PDF 批注，而是：

- 强对话。
- 从用户自己的 Markdown 知识库里召回信息。
- 快速采集新的想法、链接和片段。
- 轻量查看、追问、修正。
- 把需要深度处理的任务交回桌面端。

移动端应该是 AI-first 的知识入口，而不是桌面编辑器的缩小版。

## 产品原则

移动端 Lumina 应该是：

> AI-first 的 Markdown 知识召回与采集入口。

它应该让用户随时向自己的知识库提问，看到引用来源，捕捉新信息，并在需要时回到桌面端继续深度整理。

它不应该追求桌面端完整能力，包括完整编辑器、PDF、知识图谱、插件系统、本地文件系统管理等。

## 产品形态

比“做一个客户端”更重要的产品，可能是一个云端知识 Agent。

第一阶段只面向 Markdown 知识库，不扩展到通用文件理解、Office 文档、PDF、OCR、图片识别或“上传任何文件”的场景。先把 Markdown 的召回、问答和 Agent 工作流做清楚，再考虑是否扩展其他格式。

用户订阅 Lumina Cloud 后，可以获得：

- Markdown 笔记的云端存储空间。
- Markdown 解析与索引。
- 语义检索与召回。
- 每月 Agent 使用额度。
- 移动端访问入口。
- 跨设备访问自己的 Markdown 知识库。
- 与桌面端 Lumina Note 的连接或同步能力。

这个产品不应该被定位成通用网盘。如果说成“网盘”或“云盘”，用户会自然拿它和 iCloud、Google Drive、Dropbox、百度网盘比较容量、上传下载速度、文件分享、视频预览和价格。这些不是 Lumina 的优势，也会把成本和预期拉到错误方向。

更准确的定位是：

> 云端 Markdown 知识空间：用户同步或上传 Markdown 笔记，Agent 帮用户理解、检索、总结和行动。

存储只是基础设施。真正的产品价值是 Agent。

## 移动端 MVP

第一版移动端应该以对话为中心。

主流程：

- 用户向 Agent 提问。
- Agent 从 Markdown 知识库中召回相关笔记和片段。
- 回答中展示引用来源。
- 用户可以点开来源片段。
- 用户可以快速采集想法、链接、粘贴文本到 inbox。
- 用户可以把 Agent 的回答或总结保存回 Markdown 知识库。
- 用户可以把任务留给桌面端继续处理。

次级流程：

- 查看最近笔记。
- 查看 inbox。
- 进行很轻量的补充或修正。
- 查看账号、套餐和额度。

第一版明确不做：

- 完整桌面编辑器能力。
- 完整 CodeMirror 移动端编辑体验。
- 知识图谱。
- PDF 批注。
- 插件系统。
- 完整离线同步。
- 原生文件系统管理。
- 完整设置页。
- PDF、Word、PPT、表格、图片 OCR 或任意文件解析。

## 技术方向

最低成本路径应该是 Web-first：

- 先做移动 Web / PWA。
- 尽量复用现有 React、TypeScript、Zustand 和服务层代码。
- 检索、Markdown 索引和 Agent 执行放在云端或桌面 gateway 侧。
- 移动端只作为交互入口，不承担 local-first 的复杂性。

如果移动 Web 体验被验证有价值，再考虑用 Capacitor 把同一套移动 Web 包成 iOS / Android app。这条路线更接近 Obsidian 的跨端模式，而不是继续维护 SwiftUI + Kotlin 两套原生客户端。

当前 `mobile/` 原生方向不作为后续主线。

## 小程序定位

微信小程序不应作为主客户端。

它可以作为未来的轻入口或采集渠道：

- 分享文本或链接到 Lumina。
- 快速采集内容到 inbox。
- 做简单问答。
- 绑定账号或扫码登录。

但它不应该承载核心知识空间和完整 Agent 体验。平台限制会让它看起来便宜、实际变贵。

## 商业模型

订阅应该对应真实持续成本和用户价值：

- Markdown 存储额度。
- Markdown 索引额度。
- Agent 使用额度。
- 可能的高级模型或长上下文任务。

计费单位不应只按“对话次数”。Agent 任务成本差异太大：一次简单问答和一次长上下文、多轮召回、生成结构化笔记的任务不等价。

更稳妥的是 credit 模型：

- 模型调用消耗 credit。
- 长上下文任务消耗更多 credit。
- Markdown 索引和重新索引可以消耗 credit，或受套餐限制。
- 高级模型消耗更多 credit。

可能的套餐结构：

- Free：桌面本地能力继续可用，提供很小的云端和 Agent 试用额度。
- Pro：个人 Markdown 云端知识空间、移动端访问、固定每月 credit、Markdown 召回。
- Power：更大存储、更多 Agent credit、更长任务、更大上下文和更重的知识工作流。

## 与桌面端的关系

桌面端仍然是深度工作环境：

- 长文写作。
- local-first vault 管理。
- 复杂 Agent 任务。
- PDF 工作流。
- 知识图谱和结构整理。
- 插件生态。

移动端是快速访问环境：

- 问。
- 召回。
- 采集。
- 查看。
- 交回桌面。

## 待确认问题

- 移动端是否必须绑定 Lumina Cloud，还是也支持直接连桌面 gateway？
- 桌面本地 vault 是同步到云端 Markdown 空间，还是先做独立云端知识空间？
- Markdown 召回质量达到什么标准，用户才会觉得有价值？
- 手机小屏上如何展示引用来源和上下文片段？
- Agent credit 的具体计算规则是什么？
- 免费额度给多少，既能展示价值，又不会制造不可控成本？
- 第一版是否支持语音输入？
- 采集内容只进入 inbox，还是可以直接触发 Agent 处理？

## 当前决策

当前工作方向是：

> 不做完整移动编辑器。先做 AI-first 的移动与云端 Markdown 知识 Agent 体验。以移动 Web / PWA 验证交互模型，之后再考虑 Capacitor 打包。小程序只作为未来采集或获客入口，不作为主客户端。记录基准提交 `21ba54052b698494538f74fa1bcd3660a8ecec45` 下的 `mobile/` 原生代码可废弃，不作为后续主线。
